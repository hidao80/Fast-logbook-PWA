## 1. Design Overview

### Purpose

This document records the technical design decisions and their rationale for Fast Logbook PWA. Implementation details (API and function specifications) are documented in [`docs/spec/`](spec/). Decision history (what changed, when, and why) is recorded in [`docs/ADR.md`](ADR.md). This document supplements both with the current architectural "why".

### Problem Statement

Requirements for a timestamped work-log tool:

1. **No server** ‚Äî personal data must not be transmitted externally
2. **Zero setup for end users** ‚Äî usable immediately after opening the app, without account registration
3. **Offline operation** ‚Äî continues to work when the network is disconnected
4. **Cross-device** ‚Äî supports both PC and smartphones

### Design Trade-off Summary

| Choice | Benefits | Drawbacks |
|--------|----------|-----------|
| Client-only | Privacy protection, no server cost | No cross-device sync, manual backup |
| IndexedDB (via `idb`) | Asynchronous, larger quota, doesn't block the main thread | More setup than `localStorage`; needs a one-time migration path |
| React + TypeScript + Vite | Componentized UI, type safety, HMR, optimized production build | Requires a build step; no longer runnable by opening `index.html` directly |
| `vite-plugin-pwa` (Workbox `generateSW`) | Service worker and manifest are generated and kept in sync with the asset list automatically | Less control over the exact byte-for-byte caching logic than a hand-written `sw.js` |

## 2. Architecture Principles

### 2.1 Client-Only Principle

All data processing and storage is completed within the browser. No backend server exists.

```
Browser
‚îú‚îÄ‚îÄ index.html                 # SPA shell ‚Äî <div id="root">
‚îú‚îÄ‚îÄ src/ (React + TypeScript)  # UI + logic
‚îÇ   ‚îú‚îÄ‚îÄ main.tsx                # Entry point ‚Äî HashRouter, mounts <App>/<ConfigApp>
‚îÇ   ‚îú‚îÄ‚îÄ App.tsx                 # "/" ‚Äî log CRUD, shortcuts, date selection, export, PWA install
‚îÇ   ‚îú‚îÄ‚îÄ ConfigApp.tsx            # "/config" ‚Äî settings UI
‚îÇ   ‚îú‚îÄ‚îÄ components/              # Drawer, Modal (portal-based, Bootstrap-styled)
‚îÇ   ‚îú‚îÄ‚îÄ i18n/                    # i18next setup + locale JSON
‚îÇ   ‚îî‚îÄ‚îÄ lib/                     # storage.ts, download.ts, utils.ts
‚îú‚îÄ‚îÄ IndexedDB ("fast-logbook-pwa" DB, "kv" store)  # Data persistence
‚îú‚îÄ‚îÄ Service Worker (generated by vite-plugin-pwa)  # Offline cache
‚îî‚îÄ‚îÄ Cache API                                       # Asset cache
```

**Rationale**: Work logs may contain personal or business information. Eliminating external transmission at the architecture level reduces privacy risk to zero by design. This principle is unchanged since Version 2; what changed is *how* data is stored and *how* the UI is built (see below).

### 2.2 Build Principle (superseded 2026-06-13)

> **Version 2 said**: "Opening `index.html` directly in a browser is sufficient to run the application. No TypeScript compilation or npm bundling is required."
>
> **This is no longer true.** As of the React/TypeScript/Vite rewrite (ADR-012 in [`docs/ADR.md`](ADR.md)), a build step is mandatory.

**Current state**:

```bash
npm run dev     # Vite dev server (HMR) at http://localhost:3000
npm run build   # Production build ‚Üí dist/
npm run preview # Serve the production build locally
```

**Rationale for the change**: React's component model and TypeScript's type checking provide enough maintainability and correctness benefit (especially as `App.tsx` grew to handle log CRUD, date roll-over, shortcuts, cross-tab sync, and PWA install state) to outweigh the cost of introducing a build step. `vite-plugin-pwa`'s `generateSW` strategy also removes the previously hand-maintained `assets` array in `sw.js` (a known source of bugs in Version 2 ‚Äî see ¬ß11.1 of that revision), since Workbox derives the precache list from the build output.

**Consequence**: `npm ci` (or `npm install`) followed by `npm run build` is now required before the app can be served, including in Docker (see [¬ß8.2](#82-docker--nginx-configuration)) and Netlify (see [¬ß8.1](#81-static-hosting-netlify)) deployments.

### 2.3 Module Structure

```
src/
‚îú‚îÄ‚îÄ main.tsx        # Entry point: mounts the HashRouter with "/" and "/config" routes
‚îú‚îÄ‚îÄ App.tsx         # Main page: log CRUD, shortcuts, date roll-over, export, PWA install
‚îú‚îÄ‚îÄ ConfigApp.tsx   # Settings page: rounding unit, shortcuts, roll-over time
‚îú‚îÄ‚îÄ globals.d.ts    # Ambient types (e.g. BeforeInstallPromptEvent)
‚îú‚îÄ‚îÄ components/
‚îÇ   ‚îú‚îÄ‚îÄ Drawer.tsx  # Bootstrap offcanvas side menu, rendered via createPortal
‚îÇ   ‚îî‚îÄ‚îÄ Modal.tsx   # Bootstrap modal dialog, rendered via createPortal
‚îú‚îÄ‚îÄ i18n/
‚îÇ   ‚îú‚îÄ‚îÄ index.ts            # i18next + react-i18next initialization, browser language detection
‚îÇ   ‚îî‚îÄ‚îÄ locales/{en,ja}.json
‚îî‚îÄ‚îÄ lib/
    ‚îú‚îÄ‚îÄ storage.ts   # IndexedDB key-value wrapper (via `idb`) + localStorage migration
    ‚îú‚îÄ‚îÄ download.ts  # Log parsing, time calculation, formatted-log HTML/Markdown export
    ‚îî‚îÄ‚îÄ utils.ts     # Date/time helpers, HTML escaping, theme, PWA install wiring
```

Dependencies flow one-way: `App.tsx` / `ConfigApp.tsx` / `components/` ‚Üí `lib/`. Modules inside `lib/` do not depend on each other.

**Routing**: `main.tsx` uses `createHashRouter` (hash-based routing, e.g. `/#/config`) rather than browser history routing. This avoids needing server-side rewrite rules for client-side routes on static hosts ‚Äî though Netlify's `[[redirects]]` rule (`/* ‚Üí /index.html`, see ¬ß8.1) would also handle history-based routing if it were adopted later.

**Cross-tab sync**: Both `App` and `ConfigApp` open a `BroadcastChannel('fast-logbook-sync')` and post `{ key, value }` messages when settings (rounding unit, shortcuts, roll-over time) or log data change, so multiple open tabs/windows stay in sync without polling IndexedDB.

## 3. PWA Design

### 3.1 Web App Manifest

The manifest is no longer a static `manifest.json` file in the repository; it is generated by `vite-plugin-pwa` from the `manifest` object in [`vite.config.js`](../vite.config.js) (see ADR-016 in `docs/ADR.md`):

| Field | Value | Rationale |
|-------|-------|-----------|
| `display` | `standalone` | Hides browser UI for a native-app-like experience |
| `start_url` | `/index.html` | Specifies HTML explicitly rather than root `/` |
| `id` | Fixed UUID string | Stabilizes the identity of the installed PWA |
| `theme_color` | `#62BF04` | Brand color shown in address bar and taskbar |
| `version` | `26.06.13` (hardcoded in `vite.config.js`) | Intended as the cache-busting version string ‚Äî see [¬ß7.4](#74-known-version-string-mismatch) for a known drift issue |
| `lang` | `ja` | Default manifest language |

Icons are provided in 6 sizes from 48px to 512px (`img/android-launchericon-*.png`), unchanged from Version 2.

### 3.2 Service Worker Strategy

The hand-written `sw.js` (Stale-While-Revalidate + `Date` header comparison) described in Version 2 has been **replaced** by a Workbox service worker auto-generated by `vite-plugin-pwa` (`strategies: 'generateSW'`, output filename `sw.js`).

```javascript
// vite.config.js
VitePWA({
  registerType: 'autoUpdate',
  strategies: 'generateSW',
  filename: 'sw.js',
  workbox: {
    globPatterns: ['**/*.{js,css,html,png,ico,svg,woff2,json}'],
  },
  devOptions: { enabled: true },
})
```

**Update behavior** (see ADR-014 in `docs/ADR.md` for the iteration history): `App.tsx` registers the service worker and reloads the page **only on a controller change after the service worker already had a controller** ‚Äî i.e., only on an actual update, not on first install:

```javascript
if ('serviceWorker' in navigator) {
  if (navigator.serviceWorker.controller) {
    navigator.serviceWorker.addEventListener('controllerchange', () => {
      window.location.reload();
    });
  }
  navigator.serviceWorker.register('sw.js');
}
```

`devOptions.enabled: true` makes the service worker active during `npm run dev` as well, so update/caching behavior can be verified without a production build.

**Rationale for switching to Workbox**: The Version 2 design carried a known constraint ‚Äî the `assets` array in `sw.js` had to be updated by hand whenever a new file was added, and a real bug existed where the fetch handler wrote to a hardcoded `'my-cache'` key instead of `CACHE_NAME`. `generateSW` derives the precache manifest from the Vite build output automatically, eliminating both classes of bug.

### 3.3 Offline Support for Third-Party Libraries

Bootstrap and Bootstrap Icons are no longer loaded from a CDN at runtime. They are installed as npm dependencies (`bootstrap`, `bootstrap-icons`) and imported directly in `main.tsx`:

```typescript
import 'bootstrap/dist/css/bootstrap.min.css';
import 'bootstrap-icons/font/bootstrap-icons.css';
```

Vite bundles them into the build output, so they are automatically included in the Workbox precache glob (`**/*.{js,css,...}`) without any manual `assets` list maintenance. This removes the CDN-outage risk and the SRI-hash-maintenance burden that applied to the main app shell in Version 2 (CDN + SRI is still used for the **exported HTML file**'s Bootstrap/Font Awesome includes ‚Äî see [¬ß5.2](#52-output-format)).

### 3.4 PWA Installation Flow

```
beforeinstallprompt fires
    ‚Üì
event.preventDefault()
    ‚Üì
Store the event in a React ref (deferredPromptRef)
    ‚Üì
setInstallBtnVisible(true) ‚Üí button becomes visible
    ‚Üì
User clicks ‚Üí deferredPromptRef.current.prompt()
    ‚Üì
userChoice resolves ‚Üí hide button
    ‚Üì
(or) appinstalled fires ‚Üí hide button, clear the ref
```

Compared to Version 2's DOM-attribute-based approach (`elem.promptEvent`, toggling a `d-none` class), the install prompt event is now held in a `useRef` and visibility is driven by React state (`installBtnVisible`). An explicit `appinstalled` listener was added (ADR-017 in `docs/ADR.md`) to hide the button immediately when installation completes, rather than relying solely on the `userChoice` promise from the prompt flow.

As before, the install button only appears in environments where `beforeinstallprompt` fires (Chromium-based browsers); it stays hidden on iOS Safari and other unsupported browsers.

## 4. Log Format Design

*(Unchanged from Version 2 ‚Äî re-implemented in TypeScript in `src/lib/download.ts` / `src/App.tsx`, with identical on-disk format for backward compatibility with existing exported logs.)*

### 4.1 Format Specification

```
YYYY-MM-DD HH:MMCategory;Detail\n
```

| Element | Specification | Rationale |
|---------|--------------|-----------|
| Datetime format | `YYYY-MM-DD HH:MM` | ISO 8601 compliant, sortable |
| Separator | `;` | Visually distinguishes category from sub-detail |
| Between datetime and category | No space | Datetime is extractable via fixed-length 16 chars (`slice(0, 16)`) |
| Between entries | `\n` | Human-readable plain text format |

### 4.2 Special Prefixes

| Prefix | Meaning | Example |
|--------|---------|---------|
| `^` | Excluded from actual work time | `^Lunch`, `^Break` |
| `@` | Context (convention, optional) | `@work`, `@personal` |
| `+` | Project name (convention, optional) | `+ProjectA` |

Only `^` is interpreted by the application. `@` and `+` are user conventions and are not enforced.

### 4.3 Time Calculation Algorithm

Implemented in `parse()` in `src/lib/download.ts`. The time difference between two consecutive entries is calculated and attributed to the category of the earlier entry, exactly as in Version 2:

```
Entry 1: 2025-12-13 09:00 ProjectA;Meeting
Entry 2: 2025-12-13 10:30 ProjectB;Coding
Entry 3: 2025-12-13 12:00 ^Lunch

‚Üí ProjectA time = 10:30 - 09:00 = 90 min
‚Üí ProjectB time = 12:00 - 10:30 = 90 min
‚Üí Lunch excluded from actual work time
```

**Midnight crossing**: when the hour difference is negative, 24 hours is added to correct the calculation.

**Rounding**: `Math.floor(minutes/60) + Math.round(minutes%60/roundingUnit) * roundingUnit/60`, unchanged.

### 4.4 Per-Day View and Date Roll-Over (new in Version 3)

`App.tsx` now scopes the visible textarea to a single logical "day", rather than showing the entire log at once:

- A **roll-over time** (`date-roll-over-time`, default `05:00`) defines when a logical day starts. If the current wall-clock time is before the roll-over time, the logical date is "yesterday".
- `getDateBoundaries(selectedDate, rollOver)` computes a `[start, end)` window of exactly 24 hours starting at `rollOver` on `selectedDate`.
- Editing is buffered in `localStorage` (`log_buffer` / `log_buffer_date`) while a day is open, then merged back into the full IndexedDB log (`flushBuffer()`) on tab switch, visibility change, or date change ‚Äî merge is a sort-and-replace keyed by the 16-character timestamp prefix, not an append, so edits to already-logged lines are respected.
- A first-run notice modal (`isDateNoticeModalOpen`, gated by `notice_date_selector`) explains the new per-day view to users upgrading from the old single-textarea behavior.

**Rationale**: as logs accumulate over months, loading and rendering the entire log in one `<textarea>` degrades both usability (scrolling to "today") and performance. Scoping to a logical day, with an explicit roll-over time, matches how the format's `^Lunch`/category conventions are typically used (one working day at a time) while still storing one contiguous log in IndexedDB.

## 5. Export Pipeline Design

### 5.1 Indirect Download Mechanism

The download mechanism is conceptually unchanged from Version 2, but the hand-off no longer goes through `localStorage` ‚Äî it uses module-level variables in `src/lib/download.ts` (ADR-010 in `docs/ADR.md`), since the download is now always triggered and consumed within the same module/page lifecycle:

```
Call download(html)
    ‚Üì
Generate Blob ‚Üí Object URL
    ‚Üì
Store { _downloadUrl, _downloadFilename } as module-level variables
    ‚Üì
Dispatch startDownload CustomEvent on window
    ‚Üì
window-level event listener (registered once at module load) receives it
    ‚Üì
Create <a> element dynamically ‚Üí click() ‚Üí remove
    ‚Üì
URL.revokeObjectURL() to free memory
    ‚Üì
Clear the module-level variables
```

**Rationale for moving off `localStorage`**: this hand-off no longer needs to survive a `window.open()` boundary or a separate `config.html` page (those were removed in the React rewrite), so the synchronous, simpler module-variable pattern is sufficient and avoids an unnecessary IndexedDB/localStorage round-trip per download.

### 5.2 Output Format

Unchanged: a single self-contained HTML file bundles 3 formats, generated by `generateFormattedLog()`:

```html
<!-- Section 1: HTML summary table -->
<table id="html_summary-source">...</table>

<!-- Section 2: Plain text (original log) -->
<pre><code id="plaintext_log-source">...</code></pre>

<!-- Section 3: Markdown table -->
<pre><code id="markdown_summary-source">...</pre>
```

The exported HTML still loads Bootstrap and Font Awesome from a CDN (cdnjs) with SRI `integrity` attributes ‚Äî this is intentionally **not** bundled by Vite, since the export is a standalone artifact meant to be opened independently (e.g. emailed) rather than served by this app's own origin. Each section has a clipboard-copy affordance gated by the Clipboard API permission state.

## 6. Internationalization (i18n) Design

### 6.1 Engine Replacement: `Multilingualization` ‚Üí i18next / react-i18next

Version 2's custom `Multilingualization` static class (DOM-attribute scanning + `MutationObserver` for flash prevention) has been replaced by `i18next` + `react-i18next` (ADR-013 in `docs/ADR.md`). React's render cycle naturally avoids the translation-flash problem that motivated the `MutationObserver` approach in the vanilla-JS version, since translated strings are rendered as part of the initial component output rather than patched into the DOM after the fact.

```typescript
// src/i18n/index.ts
i18next.use(initReactI18next).init({
  resources: { en: { translation: en }, ja: { translation: ja } },
  lng: detectedLang,
  fallbackLng: 'en',
  interpolation: { escapeValue: false },
});
```

### 6.2 Translation Key Design

Translation resources now live in `src/i18n/locales/{en,ja}.json` (133 keys each, as of this writing) rather than being embedded in a JS dictionary object. Components call `useTranslation()` and reference keys via `t('key_name')`; non-component code (e.g. `download.ts`) imports the configured `i18next` instance directly and calls `i18next.t(...)`.

`data-translate` HTML attributes are no longer used ‚Äî translation is expressed through JSX/TSX, consistent with React idioms.

### 6.3 Language Detection

```typescript
const detectedLang = (() => {
  const lang = (navigator.languages?.[0] ?? navigator.language ?? 'en').slice(0, 2);
  return lang in { en: 1, ja: 1 } ? lang : 'en';
})();
```

Behaviorally equivalent to Version 2 (Japanese environments use `ja`; everything else falls back to `en`), but now checks against the actual set of available resource languages instead of a single `startsWith('ja')` check.

## 7. Toolchain Design

### 7.1 Biome (lint + format)

`biome.json`'s lint/format scope moved from `js/**/*.js` to `src/**/*.ts` and `src/**/*.tsx`, reflecting the TypeScript rewrite. The `noStaticOnlyClass` exception (needed in Version 2 to preserve the static-only `Multilingualization` class) is no longer present, since that class was deleted along with the rest of the vanilla-JS `js/lib/` tree. A new exception was added instead:

```jsonc
{
  "linter": {
    "rules": {
      "security": {
        "noDangerouslySetInnerHtml": "off"  // copy-button script in generateFormattedLog()
      }
    }
  }
}
```

`noUnusedVariables: "warn"` is retained from Version 2.

### 7.2 TypeScript Configuration

`tsconfig.json` targets `ES2020`, uses `"moduleResolution": "bundler"` (Vite-native resolution, no `.js` extension rewriting needed) and `"jsx": "react-jsx"` (the React 17+ automatic JSX runtime, so component files don't need an explicit `import React` for JSX). `strict: true` and `noUnusedLocals: true` are both enabled.

### 7.3 CI/CD (GitHub Actions)

Workflow structure is unchanged in shape from Version 2 (`lint.yml`, `audit.yml`), but `lint.yml` now runs Biome against `src/` instead of `js/`. The Takumi Guard supply-chain scan and `pnpm`-based audit step (added 2026-03-13, see ADR-008 in `docs/ADR.md`) predate the React rewrite and are unaffected by it.

### 7.4 Known Version-String Mismatch

As of this writing, `package.json`'s `"version"` field (`26.02.19`) and the PWA manifest `version` field hardcoded in `vite.config.js` (`26.06.13`) have drifted apart. In Version 2, the manifest `version` was the single source of truth used both as the displayed app version and as the Service Worker cache-busting key. With the move to `vite-plugin-pwa`/Workbox (¬ß3.2), cache busting is now handled by Workbox's own revisioned precache manifest, so the `version` field in `vite.config.js` is effectively cosmetic ‚Äî but `ConfigApp.tsx` and `App.tsx` both still fetch `/manifest.json` at runtime to display "the app version" to the user, which means the UI currently shows the (stale) `vite.config.js` value rather than `package.json`'s. This is a known inconsistency to resolve (e.g. by generating one from the other at build time), not an intentional design decision.

## 8. Deployment Design

### 8.1 Static Hosting (Netlify)

Unlike Version 2 ("no build process, files are served directly on `git push`"), Netlify now runs an actual build (ADR-015 in `docs/ADR.md`):

```toml
# netlify.toml
[build]
  command = "npm run build"
  publish = "dist"

[[headers]]
  for = "/manifest.json"
  [headers.values]
    Content-Type = "application/manifest+json"

[[redirects]]
  from = "/*"
  to = "/index.html"
  status = 200
```

The SPA fallback redirect (`/* ‚Üí /index.html`, status 200) is defensive: the app currently uses `createHashRouter` (¬ß2.3), which doesn't strictly require this rule since route state lives in the URL fragment, but the rule is harmless and protects against a future switch to browser-history routing.

### 8.2 Docker / Nginx Configuration

The Dockerfile's build stage is now a real build, not a conditional no-op:

```dockerfile
FROM node:24.11.1-bookworm-slim AS builder
WORKDIR /app
COPY package*.json ./
RUN if [ -f package-lock.json ]; then npm ci; elif [ -f package.json ]; then npm install; fi
COPY . .
RUN if grep -q "\"build\"" package.json; then npm run build; fi

FROM nginx:alpine
# ...copies /app (with dist/ promoted to webroot if present)...
```

The "promote `dist/` to webroot if present" step is a defensive carry-over from when the app could also run unbuilt; with the build now mandatory, `dist/` is always present, but the conditional logic is harmless and avoids needing two near-duplicate Dockerfiles in the meantime.

Nginx cache headers are unchanged from Version 2 (`sw.js`/`index.html` ‚Üí `no-cache`; static assets ‚Üí `immutable, 1y`), since this cache policy is about HTTP delivery, not about the build pipeline.

## 9. Security Design

### 9.1 XSS Protection

`escapeHtml()` (now in `src/lib/utils.ts`, identical implementation to Version 2) is still required before interpolating user-entered log data into HTML ‚Äî used in `toHtml()` and `generateFormattedLog()` in `src/lib/download.ts`. React's JSX escaping protects the live app UI by default (text content is never interpolated as raw HTML in `App.tsx`/`ConfigApp.tsx`), but the **exported HTML file** is built as a plain string, not JSX, so `escapeHtml()` remains load-bearing there.

### 9.2 Content Security Policy

Still not configured (same known gap as Version 2). Since Bootstrap/Bootstrap Icons are now bundled by Vite rather than loaded from a CDN (¬ß3.3), a CSP for the **app shell** would actually be simpler to write today (`script-src 'self'` would mostly suffice) ‚Äî the CDN dependency now only exists in the **exported HTML artifact** (¬ß5.2), which is opened as a standalone file outside this app's origin and isn't covered by this app's CSP regardless.

### 9.3 IndexedDB Privacy

- Storage is scoped to the same origin, same as `localStorage` was.
- No data is ever transmitted to a server.
- The GA4 analytics module (`js/lib/analytics.js` in Version 2) has been **removed entirely** (ADR-005 in `docs/ADR.md`) ‚Äî there is currently no telemetry of any kind, app-interaction or otherwise.

### 9.4 Console Disabling in Production

The `$$disableConsole()` call (via `indolence.min.js`) described in Version 2 was removed along with the rest of the vanilla-JS `js/lib/` tree. No equivalent mechanism currently exists in the React app; console output suppression in production is an open item (see ¬ß11.2).

### 9.5 Hardcoded-String Bug (fixed)

Version 2's security notes flagged a known bug: the storage-quota-exceeded error message was a hardcoded Japanese string bypassing the i18n system. This has been fixed ‚Äî `storage_quota_exceeded` is now a proper i18next key present in both `en.json` and `ja.json`, and `App.tsx`'s `saveLogs()` calls `t('storage_quota_exceeded')` when catching a `QuotaExceededError`.

## 10. Performance Design

### 10.1 Debounced Saves

The 300ms debounce pattern from Version 2 is retained in `App.tsx`'s `handleTextareaInput`, now implemented with a `useRef<ReturnType<typeof setTimeout>>` instead of a module-level variable. On IME (Japanese input) composition end, an immediate save is still forced (`handleTextareaCompositionEnd`) to avoid losing input while a debounce is pending. Saves write to `localStorage` (the per-day edit buffer, ¬ß4.4) synchronously; the buffer is only merged into IndexedDB (an async operation) on tab-hide or date change, keeping keystroke-path writes fast.

### 10.2 Speculation Rules API

Unchanged from Version 2 ‚Äî `index.html` still embeds a `<script type="speculationrules">` block for prerender/prefetch hints on Chrome 108+, silently ignored elsewhere.

### 10.3 Font and Icon Optimization

Unchanged: Bootstrap Icons (CSS font) over individual SVGs; no custom `font-family`, relying on the OS default font stack.

### 10.4 Build-Time Optimizations (new in Version 3)

Vite's production build (`npm run build`) provides tree-shaking, code-splitting, and asset hashing that did not exist in the no-bundler Version 2 setup. `vite.config.js` explicitly dedupes and pre-bundles `react`, `react-dom`, `react-router`, and `react-router-dom` (`resolve.dedupe` / `optimizeDeps.include`) to avoid duplicate copies of these packages being pulled in via transitive dependencies during dev-server cold starts.

## 11. Known Constraints & Future Improvements

### 11.1 Current Constraints

| Constraint | Impact | Workaround |
|------------|--------|-----------|
| Build step required | App can no longer be run by opening `index.html` directly; `npm run build`/`npm run dev` is mandatory | Document clearly in `CLAUDE.md` / `README.md` (already done) |
| `package.json` version vs. manifest `version` drift (¬ß7.4) | Displayed app version may not match the actual release | Generate one from the other at build time |
| No cross-device sync | Cannot share logs across devices | Manual export ‚Üí import |
| No CSP configured | Weaker defense against injection attacks for the app shell | Add CSP headers (now simpler ‚Äî see ¬ß9.2) |
| No production console suppression | Console output is not silenced in production builds (Version 2's `$$disableConsole()` was removed, no replacement added) | Reintroduce an equivalent guard, or rely on Vite's production `console.log` stripping if configured |
| Per-day log buffer (`log_buffer` in `localStorage`) and the full log (IndexedDB) are two separate stores reconciled by `flushBuffer()` | A crash between buffer write and flush could theoretically leave edits stuck in the buffer | `flushBuffer()` runs on `visibilitychange`/date-change/load, minimizing the window |

### 11.2 Future Improvement Candidates

- **Resolve the version-string drift** (¬ß7.4): derive the manifest `version` from `package.json` at build time instead of hardcoding it in `vite.config.js`.
- **Reintroduce production console suppression**, now via a Vite build option (e.g. stripping `console.*` calls) rather than a runtime DOM utility.
- **Add CSP headers**: now that third-party scripts are bundled rather than CDN-loaded for the app shell itself, a strict `script-src 'self'` policy is feasible.
- **Data compression**: still an open candidate ‚Äî `CompressionStream` API to shrink the stored log, now relevant to IndexedDB storage rather than `localStorage` quota.
- **Import functionality**: UI to re-import logs from a previously exported HTML file ‚Äî still not implemented.
- **Browser-history routing**: if ever desired, the Netlify SPA-fallback redirect (¬ß8.1) and Nginx `try_files` rule (¬ß8.2) already support it; only `main.tsx`'s `createHashRouter` ‚Üí `createBrowserRouter` switch would be needed.
